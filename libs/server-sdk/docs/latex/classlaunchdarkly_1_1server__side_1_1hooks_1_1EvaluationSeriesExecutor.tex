\doxysubsection{Detailed Description}
Executes hooks for a single evaluation series. Each instance handles one evaluation from before\+Evaluation through after\+Evaluation.

Handles exceptions/errors from hooks and logs them without affecting the evaluation operation.

THREADING AND LIFETIME GUARANTEES\+:
\begin{DoxyItemize}
\item All hook callbacks are executed SYNCHRONOUSLY in the same thread as the caller of the SDK method (e.\+g., Bool\+Variation, Track).
\item \mbox{\hyperlink{classlaunchdarkly_1_1server__side_1_1hooks_1_1Hook}{Hook}} callbacks complete before control returns to the caller.
\item \mbox{\hyperlink{classlaunchdarkly_1_1Context}{Context}} objects (\mbox{\hyperlink{classlaunchdarkly_1_1server__side_1_1hooks_1_1EvaluationSeriesContext}{Evaluation\+Series\+Context}}, \mbox{\hyperlink{classlaunchdarkly_1_1server__side_1_1hooks_1_1TrackSeriesContext}{Track\+Series\+Context}}) passed to hooks are stack-\/allocated and remain valid for the entire duration of the callback.
\item This synchronous execution model is CRITICAL for the C bindings, which return pointers to internal data that are only valid during the callback. Asynchronous execution would create dangling pointers.
\item \mbox{\hyperlink{classlaunchdarkly_1_1server__side_1_1hooks_1_1Hook}{Hook}} implementations MUST NOT store references, pointers, or string\+\_\+views from context objects beyond the immediate callback execution.
\end{DoxyItemize}

IMPLEMENTATION REQUIREMENTS\+:
\begin{DoxyItemize}
\item Do NOT introduce async hook execution without redesigning the C bindings.
\item Do NOT move hook execution to background threads.
\item If async hooks are needed in the future, the C bindings must be changed to return heap-\/allocated copies instead of pointers to stack data. 
\end{DoxyItemize}
